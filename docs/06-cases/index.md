# 사례에서 배우는 AI-Ready Data

사례는 제품 선택의 증거가 아니라 문제를 어떻게 좁히고 데이터를 어떤 통제로 바꿨는지
배우는 자료다. 벤더 고객 사례의 수치는 해당 조직과 벤더가 발표한 값으로 표시하며,
독립 검증이나 우리 조직의 예상 성과로 오해하지 않는다.

## 사례 읽는 다섯 질문

1. 사용자의 어떤 반복 업무와 결정을 바꿨는가?
2. 원천 데이터는 무엇이고 어떤 품질 문제가 있었는가?
3. 표준화·권한·출처·사람 검토를 어떻게 설계했는가?
4. 모델이 아니라 데이터·검색·운영에서 무엇을 측정했는가?
5. 공개되지 않았거나 우리 조직에서 다시 검증할 가정은 무엇인가?

## 반복해서 나타나는 패턴

- “전사 AI”보다 문서 검토, 검색, 절차 안내처럼 경계가 있는 업무에서 시작한다.
- 원천을 파싱해 바로 생성하지 않고 분류·메타데이터·검색·검토 단계를 둔다.
- 고유한 업무 기준과 예외를 데이터로 명시한다.
- 출처와 사람 검토는 정확도 수치 뒤에 붙는 옵션이 아니다.
- 공개된 기술 구조보다 운영 권한·보존·삭제·실패 비용이 덜 공개되는 경우가 많다.

국내 공공기관의 최신 도입 절차와 고려사항은
[NIA 공공부문 AI 도입·활용 가이드](https://www.nia.or.kr/site/nia_kor/ex/bbs/View.do?bcIdx=29526&cbIdx=37989)에서
확인한다. 일반 기업은 그대로 준수 대상으로 받아들이기보다 국내 환경의 체크포인트와
사례 출처로 활용한다.

!!! info "사례 표시 규칙"
    `verified-case`는 조직·벤더·정부기관 등 확인 가능한 원출처가 있다는 뜻이지 모든
    주장이 독립 검증됐다는 뜻이 아니다. `composite-case`는 여러 현장의 공통 문제를
