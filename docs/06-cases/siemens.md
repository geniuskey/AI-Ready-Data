# Siemens의 분산 제품 정보 검색

<span class="case-badge verified-case">verified-case · 공개 출처 사례</span>

## 문제

Siemens 고객은 여러 사업부와 전자상거래를 포함한 15~20개 사이트에서 기술 정보를
찾아야 했다. 마케팅 페이지와 기술 문서가 섞이고 제품·서비스 데이터가 복잡해 사용자가
원하는 사양을 찾기 어려웠다.

## 공개된 접근

AWS의 Siemens 고객 사례는 자연어 질의를 다음 단계로 처리했다고 설명한다.

1. 질의의 유효성과 답변 가능한 지식베이스를 검사한다.
2. 사용자 의도를 분류해 적절한 검색 백엔드로 라우팅한다.
3. 해당 지식베이스에서 표적 검색을 수행한다.
4. 결과를 요약하고 금지 주제·타사 제품·법적 위험 질의를 가드레일로 검사한다.

원출처는 새 검색이 무결과 검색을 제거하고 검색 속도를 300% 개선했으며 사용자 만족도가
11% 증가했다고 발표한다. 또한 프로젝트의 가드레일 작업이 전사 생성형 AI 법적 위험
거버넌스 틀로 이어졌다고 설명한다.
([AWS Siemens 고객 사례](https://aws.amazon.com/solutions/case-studies/siemens-nova-case-study/))

## AI-Ready Data 관점의 교훈

### 1. 하나의 거대한 검색보다 원천 지도를 만들었다

질의를 분류하고 적합한 백엔드로 보내는 구조는 원천마다 목적·범위·권위·스키마가
다름을 인정한다. 전사 파일을 무차별로 한 색인에 넣는 것과 다르다.

### 2. “찾을 수 없음”도 제품 동작이다

원하는 제품이 더 이상 Siemens 소관이 아니라면 적절한 다음 경로를 안내한다. 근거가
없을 때 그럴듯한 답을 만드는 대신 범위 밖임을 설명하는 것이 신뢰를 만든다.

### 3. 데이터 경계와 법적 경계를 함께 라우팅한다

검색 품질만이 아니라 타사 제품, 주식 등 법적 위험이 있는 질문을 분류한다. 유즈케이스
정의에는 답할 원천뿐 아니라 답하지 않을 목적·주제·행동도 들어가야 한다.

## 그대로 가져오면 안 되는 것

- 공개 웹사이트 검색은 내부 기밀 문서와 ACL 요구가 다르다.
- 고객 만족·속도·비용 수치는 기존 시스템과 선택한 모델 대비 결과다.
- “무결과 검색 제거”가 모든 질문에 생성 답변을 했다는 뜻으로 해석돼서는 안 된다.
- 서비스별 백엔드와 지식베이스 품질 관리의 상세 운영은 공개 자료만으로 알 수 없다.

## 우리 조직에 적용할 실험

- 현재 검색 로그에서 무결과·재검색·이탈 질문을 100개 수집한다.
- 질문을 원천/업무 도메인/권한/답변 가능/명확화 필요/범위 밖으로 분류한다.
- 단일 벡터 검색과 원천 라우팅+하이브리드 검색을 비교한다.
- 정답률뿐 아니라 적절한 범위 밖 안내와 잘못된 원천 사용률을 측정한다.
- 법무·보안이 금지 주제와 표현을 릴리스 전에 골든셋에 추가한다.

<p class="source-note">출처와 수치는 AWS 고객 사례 및 Siemens 진술에 기반한다. 공개 웹 검색 사례를 내부 보안 구조로 그대로 복제하지 않는다.</p>
