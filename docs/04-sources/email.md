# 이메일을 기록과 지식으로 전환하는 플레이북

메일함에는 요청, 결정, 근거, 첨부와 사람의 관계가 함께 있지만 그대로는 좋은 지식베이스가
아니다. 개인 대화·서명·반복 인용·자동 알림·낡은 첨부·민감정보도 함께 있기 때문이다.
[NARA 이메일·전자메시지 관리](https://www.archives.gov/records-mgmt/email-mgmt)는 이메일도
기록 생애주기와 메타데이터, 보존 규칙 안에서 관리해야 함을 보여준다. 국내 조직은
자체 기록관리·개인정보·노무·법적 보존 정책을 먼저 적용한다.

::: info 두 준비축을 차례로 적용한다
스레드·첨부·Message-ID·ACL을 보존하고 본문을 추출하는 일은 **Data Ready**, 제안과
최종 결정·근거·효력을 사람이 판정하는 일은 **Knowledge Ready**다. 한 번의 요약
작업으로 두 단계를 합치지 않는다.
:::

## 첫 질문은 “무엇을 추출할까”가 아니다

다음을 먼저 결정한다.

- 어떤 업무 계정과 공유 메일함을 어떤 기간에 수집하는가?
- 개인·사적·노조·고충·법률 자문 등 제외 또는 강화 통제가 필요한 범위는 무엇인가?
- 메일과 첨부의 소유자, 보존기간, 법적 보존 명령은 무엇인가?
- AI가 지원할 업무 질문은 무엇이며 답하면 안 되는 질문은 무엇인가?
- 누가 원문·추출 사실·요약·관계 정보를 각각 볼 수 있는가?

::: danger 전 직원 메일을 한 번에 색인하지 않는다
메일 검색 권한과 AI가 답변을 조합하는 권한은 같지 않다. 목적·기간·계정·폴더·정보
유형을 좁힌 파일럿부터 시작한다.
:::

## 수집해야 할 단위

| 계층 | 보존할 내용 |
| --- | --- |
| 메시지 | Message-ID, 발신/수신/참조, 발송시각, 제목, 본문, 중요도 |
| 스레드 | In-Reply-To/References, 대화 ID, 시간 순서, 누락 여부 |
| 첨부 | 파일명, 실제 형식, 해시, 메시지 연결, 버전 후보, 악성 검사 |
| 권한·정책 | 원 메일함/폴더, 사용자·그룹, 정보등급, 보존·법적 보존 |
| 변환 | 파서 버전, 인용문 제거, 서명 탐지, 언어, 품질 플래그 |

BC/BCC와 배포 목록의 의미, 대리 발송, 공유 메일함, 일정 초대도 정책에 맞게 다룬다.
Apache Tika는 메일 형식도 지원하지만
([지원 형식](https://tika.apache.org/2.9.2/formats.html)), 실제 보존 형식과 메일 서버
API의 메타데이터를 함께 검증해야 한다.

## 스레드를 정리하는 순서

1. Message-ID와 참조 헤더로 스레드를 구성하고 시간만으로 추측하지 않는다.
2. 이전 답장 인용, 자동 서명, 법적 고지, 추적 픽셀을 별도 영역으로 분리한다.
3. 첨부는 해시로 중복을 찾되 같은 파일명이 같은 내용이라고 가정하지 않는다.
4. 메시지별 새로 추가된 내용을 보존하고 스레드 전체를 원문으로 돌아갈 수 있게 한다.
5. 사람·조직·프로젝트·문서 ID는 승인된 사전으로 정규화하고 불확실성을 표시한다.
6. 결정·요청·약속·기한·근거를 **후보**로 추출한다.
7. 업무 소유자가 승인한 항목만 의사결정 로그나 업무 시스템에 승격한다.

## AI가 추출할 수 있는 것과 승인할 것

| AI 후보 추출 | 사람이 확인할 질문 |
| --- | --- |
| “A안으로 진행” 결정 후보 | 최종 결정권자의 확정 메시지인가? 이후 번복됐는가? |
| 담당자·기한 | 요청인지 수락인지, 기한이 어느 시간대인지 명확한가? |
| 첨부 최신본 후보 | 승인·효력·대체 관계가 문서 안에 있는가? |
| FAQ 후보 | 일회성 상황인가, 반복 사용이 허용된 지식인가? |
| 관계·전문가 후보 | 조직도·업무 역할로 정당한가, 사적 관계를 추론하지 않는가? |

사람의 감정, 성과, 성향, 민감한 관계를 메일에서 추론해 프로파일링하지 않는다. 유즈케이스
목적에 필요한 최소 정보만 처리한다.

## 지식으로 승격하는 다섯 줄

```markdown
### 결정 DEC-2026-014
- 결정: 국내 공장 정비 요청은 새 A 양식을 사용한다.
- 근거: 필수 안전 필드 누락률 비교와 운영책임자 승인.
- 효력: 2026-08-01, 국내 공장 신규 요청부터.
- 책임: 운영지원팀이 2026-07-29까지 기본 양식을 교체한다.
- 출처: mail:shared-maintenance:<message-id>, 첨부 OPS-FORM-02 v3.0
```

추출된 지식에는 원 스레드 ID, 승인자, 승인일, 상태, 재검토일, ACL을 붙인다. 메일을
삭제하라는 뜻이 아니라 반복 사용할 결론을 명확한 기록으로 승격한다는 뜻이다.

## 검색과 요약 안전장치

- 질문자의 권한으로 검색 가능한 메시지와 첨부를 먼저 필터링한다.
- 여러 메일의 내용을 합칠 때 각 주장 옆에 메시지·시각·보낸 사람을 인용한다.
- 스레드가 불완전하거나 결정이 번복되면 “확정 불가”로 답한다.
- 이메일 주소·개인정보·비밀은 질문 목적과 권한에 따라 마스킹한다.
- 답변 캐시는 사용자·권한·원천 버전에 종속시키고 짧게 유지한다.
- 원 메일 삭제·권한 변경·법적 보존을 색인과 파생 지식에 전파한다.

## 파일럿 평가 질문

- 같은 스레드의 반복 인용을 새 결정으로 중복 집계하지 않는가?
- 가장 최근 메시지가 아니라 실제 승인된 결정을 찾는가?
- 서로 다른 첨부 버전과 번복을 사용자에게 보여주는가?
- 권한 없는 수신자·BCC·첨부 내용을 드러내지 않는가?
- 모호한 담당자·날짜·약속을 사실로 단정하지 않는가?
- 원문 메시지와 첨부를 감사 가능한 방식으로 열 수 있는가?

메일에서 나온 문서 충돌은 [EDM과 SSoT](edm-ssot.md), 결정 후보는
[의사결정 로그 템플릿](../templates/decision-log.md)으로 넘긴다.
