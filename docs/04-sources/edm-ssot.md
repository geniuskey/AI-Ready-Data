# EDM에서 SSoT를 찾고 유지하는 플레이북

SSoT(Single Source of Truth)는 파일 하나가 아니라 **어떤 질문에 어떤 원천을 권위
있다고 인정하는지에 대한 규칙과 책임**이다. 같은 업무에도 계약의 SSoT, 절차의 SSoT,
실적의 SSoT가 다를 수 있다.

::: info 이 페이지의 위치
중복·버전 후보와 메타데이터를 찾는 데까지는 **Data Ready**, 어느 원천이 언제 어디에
유효한지 사람이 판정하는 일은 **Knowledge Ready**다. SSoT는 후자의 승인 결과다.
:::

## “최신 수정일”이 진실이 아닌 이유

- 오래된 승인 문서를 최근에 열어 저장했을 수 있다.
- 최신 초안이 효력 중인 구버전보다 수정일이 늦다.
- 승인 PDF와 편집 원본의 수정일·내용이 다르다.
- 로컬 복사본은 EDM의 폐기·대체 관계를 모른다.
- 첨부파일은 메일에서 번복됐거나 다른 범위에만 적용될 수 있다.

전자기록 관리는 수집·조직·분류·저장·검색·사용·폐기라는 생애주기 능력을 요구한다.
EDM의 검색·버전 기능만으로 기록관리와 SSoT가 자동 해결되지는 않는다.
([NARA ERM 요구사항 방법론](https://www.archives.gov/records-mgmt/policy/requirements-guidance.html))

## 1단계: 저장소를 먼저 지도화한다

| 확인 영역 | 질문 |
| --- | --- |
| 공간 | 공식 저장소, 팀 폴더, 개인 폴더, 메일 첨부는 어디인가? |
| 문서 유형 | 규정, 절차, 계약, 설계, 보고, 실적의 권위 기준이 다른가? |
| 상태 | 초안·검토·승인·효력·대체·폐기를 시스템이 표현하는가? |
| 식별 | 문서 ID와 버전이 파일명 밖에도 존재하는가? |
| 권한 | 폴더 상속, 예외 ACL, 외부 공유와 다운로드 권한은 무엇인가? |
| 보존 | 보존기간, 법적 보존, 삭제·파기 증거가 있는가? |
| 연계 | 메일, 결재, 업무 시스템과 문서의 관계를 추적할 수 있는가? |

## 2단계: 중복과 버전 후보를 만든다

완전 동일 파일은 암호학적 해시로 찾고, 내용이 거의 같은 파일은 정규화 텍스트와
구조 유사도로 후보화한다. 유사도는 삭제 명령이 아니라 검토 대기열을 만드는 도구다.

- 완전 중복: 바이트 해시 동일
- 표현 중복: PDF/Word처럼 형식만 다른 동일 내용
- 파생 버전: 일부 문단·표·날짜가 변경
- 부분 중복: 여러 문서를 합친 종합본 또는 발췌본
- 충돌: 같은 범위·효력일에 서로 다른 규칙·수치

## 3단계: 권위 점수를 설명 가능하게 계산한다

AI나 규칙 엔진은 다음 증거로 SSoT **후보** 순위를 만든다.

| 증거 | 강한 신호 | 경고 신호 |
| --- | --- | --- |
| 승인 | 권한 있는 승인자와 승인 기록 | 승인자·절차 불명 |
| 상태 | 효력 중, 대체 관계 명시 | 초안, 검토중, 폐기 |
| 효력 | 질문 시점과 적용 범위 일치 | 미래 시행, 기간 만료 |
| 소유 | 지정된 업무 소유자 저장소 | 개인 폴더·메일 임시 첨부 |
| 식별 | 공식 문서 ID·버전·해시 | 파일명에만 `최종` 표기 |
| 참조 | 상위 정책·결재·업무시스템 연결 | 출처와 근거 없음 |
| 품질 | 필수 섹션·서명·단위 완전 | 페이지·표·첨부 누락 |

점수, 사용한 증거, 제외한 증거, 불확실성을 함께 보여준다. 업무 소유자는
[SSoT·충돌대장](../templates/ssot-register.md)에서 승인, 조건부 승인, 보류, 폐기,
병존 중 하나를 선택한다.

## 하나를 고르면 안 되는 경우

- 지역·제품·계약·고객에 따라 다른 문서가 동시에 유효하다.
- 과거 시점 질문에는 당시 효력 버전이 필요하다.
- 정책과 실제 업무 실적은 서로 다른 종류의 사실이다.
- 법적 분쟁이나 감사 때문에 상충 기록을 모두 보존해야 한다.

이때는 하나를 삭제하지 말고 적용 범위와 시간 조건이 있는 여러 권위 원천으로 관리한다.
SSoT보다 “신뢰 가능한 원천 집합(Authoritative Source Set)”이 정확한 표현일 수 있다.

## 4단계: 사용자에게 상태를 보이게 한다

검색과 AI 답변에 제목만 표시하지 않는다.

```text
[효력 중] OPS-MNT-014 v2.1 · 2026-07-01 시행
소유자: 설비기술팀 · 다음 검토: 2027-01-01
대체: v2.0 · 적용: 국내 A/B 공장
근거 위치: 4.2절 점검 순서
```

충돌·만료·검토중 상태는 눈에 띄게 표시하고, 답변이 어떤 시점과 적용 범위를 가정했는지
말한다.

## 5단계: 다시 무너지지 않게 한다

- 승인 저장 위치와 문서 ID 발급 규칙을 단순하게 만든다.
- 새 승인 시 이전 버전에 대체 링크와 효력 종료일을 자동 기록한다.
- 만료·검토기한·소유자 부재·동시 효력 충돌을 주기 탐지한다.
- 로컬 사본을 금지하는 대신 공식 원본으로 돌아가는 링크와 오프라인 절차를 제공한다.
- AI 답변 오류에서 발견된 문서 문제를 원천 개선 백로그로 돌린다.
- 폐기는 업무 소유자·기록관리·법적 보존 규칙의 승인을 거친다.

## 지표

- 소유자·상태·효력일이 없는 문서 비율
- SSoT 충돌 수, 평균 판정 시간, 기한 초과 수
- 폐기·만료 문서가 검색 상위에 노출된 횟수
- 공식 원본 대신 개인 복사본이 인용된 비율
- 원본 변경이 색인에 반영되는 시간
- 답변에서 적용 시점·범위를 찾을 수 있는 비율

SSoT가 정해진 뒤에는 [수집 파이프라인](../05-pipeline/ingestion.md)이 그 상태와
권한을 손실 없이 전달해야 한다.
