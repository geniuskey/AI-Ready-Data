# AI-Ready Data 준비도 진단

진단의 목적은 높은 평균 점수가 아니다. 선택한 유즈케이스를 막는 **가장 낮은
통제 수준과 모르는 사실**을 찾는 것이다. [GAO AI 책임 프레임워크](https://www.gao.gov/products/gao-21-519sp)도
AI 책임을 거버넌스, 데이터, 성능, 모니터링이라는 서로 연결된 원칙으로 다룬다.

## 먼저 진단 범위를 한 문장으로 쓴다

> `[사용자]`가 `[업무 순간]`에 `[승인된 원천]`을 근거로 `[결정/행동]`하도록 돕는다.

“전사 지식 챗봇”처럼 범위가 없는 문장은 진단할 수 없다. 예를 들어 “설비 엔지니어가
고장 조치 전에 승인된 정비 절차와 최근 유사 장애를 찾아 점검 순서를 결정하도록
돕는다”라면 사용자, 순간, 원천, 행동이 모두 보인다.

## 8개 진단축

각 항목은 증거가 있을 때만 점수를 준다. 인터뷰 답변만 있고 목록·정책·로그·샘플이
없다면 한 단계 낮게 기록한다.

| 진단축 | 핵심 질문 | 증거 예시 |
| --- | --- | --- |
| 업무 적합성 | 어떤 결정이 얼마나 나아져야 하는가? | 유즈케이스 캔버스, 기준 지표 |
| 발견 가능성 | 필요한 원천과 담당자를 찾을 수 있는가? | 데이터 인벤토리, 카탈로그 |
| 품질 | 정확·완전·최신·대표성 기준이 있는가? | 프로파일링 결과, 품질 규칙 |
| 표준화 | 용어·단위·코드·문서 구조가 합의됐는가? | 용어사전, 스키마, 문서 템플릿 |
| 소유권·거버넌스 | 승인·정정·폐기를 누가 결정하는가? | RACI, 승인·변경 기록 |
| 보안·개인정보 | 처리 근거와 최소 권한을 증명하는가? | 정보등급, ACL, 위협모델, 감사로그 |
| 출처·버전 | 답을 원본의 정확한 시점과 연결하는가? | 원본 ID, 해시, 효력일, 계보 |
| 기계 활용·운영성 | 추출·검색·삭제·모니터링을 반복하는가? | 파이프라인 로그, 골든셋, SLO |

## 0~5 성숙도

| 수준 | 이름 | 관찰되는 상태 | 다음 수준의 최소 증거 |
| --- | --- | --- | --- |
| 0 | 암묵지 | 사람·메일·개인 폴더에 의존 | 원천과 업무 질문 목록 |
| 1 | 목록화 | 무엇이 있는지는 알지만 책임·통제가 약함 | 소유자, 등급, 사용 승인 |
| 2 | 통제 | 권한·보존·책임이 정의됨 | 표준 용어·형식·품질 규칙 |
| 3 | 표준화 | 반복 가능한 구조와 검사가 있음 | 출처가 보존된 검색·인용 |
| 4 | 검색·인용 가능 | 권한 범위에서 근거 답변과 거절 가능 | 운영 SLO와 지속 평가 |
| 5 | 측정·지속개선 | 변경·삭제·오류가 측정되고 개선됨 | 다음 유즈케이스로 재사용 |

!!! warning "평균의 함정"
    보안·개인정보가 0점인데 나머지가 5점이어도 파일럿을 시작하면 안 된다.
    성숙도는 인증이 아니며, 위험한 결손을 평균으로 감추지 않는다.

## 세 개의 게이트

### 게이트 A — 수집해도 되는가

다음 중 하나라도 “아니오/모름”이면 데이터를 복사하거나 모델에 넣지 않는다.

- [ ] 유즈케이스와 허용된 사용 목적이 문서화됐다.
- [ ] 원천의 업무 소유자와 시스템 관리자가 확인됐다.
- [ ] 개인정보·영업비밀·규제정보 등 정보등급을 분류했다.
- [ ] 처리·보존·삭제 근거와 접근 승인권자가 확인됐다.
- [ ] 외부 통신, 원격 측정, 업데이트 경로를 보안 담당자가 승인했다.

### 게이트 B — 파일럿할 수 있는가

- [ ] 대표 문서와 예외·낡은 버전·충돌 문서를 포함한 샘플이 있다.
- [ ] SSoT 승인 기준과 “확정 불가” 처리 방식이 있다.
- [ ] 정답, 인용, 거절, 권한 차단을 검사할 골든셋이 있다.
- [ ] 원본과 파생물의 ID·버전·해시·ACL을 연결한다.
- [ ] 사람 검토자와 오류 접수·수정 절차가 있다.

### 게이트 C — 운영할 수 있는가

- [ ] 정확성뿐 아니라 검색 재현율, 인용 적합성, 거절 품질을 측정한다.
- [ ] 원본 변경·권한 회수·보존기간 만료가 색인과 캐시에 전파된다.
- [ ] 장애·오염·권한 누출을 탐지하고 격리·복구할 수 있다.
- [ ] 품질 SLO, 담당자, 검토 주기, 중단 기준이 합의됐다.

## 진단 워크숍 운영법

90분 워크숍에 현업 소유자, 기록·법무/개인정보, 보안, 시스템 운영, 데이터/AI
담당자가 함께 참석한다.

1. 10분: 유즈케이스 문장과 금지 범위를 합의한다.
2. 25분: 각자 [준비도 진단표](../templates/readiness-assessment.md)를 채운다.
3. 25분: 점수 차이가 2점 이상인 항목만 증거를 보며 토론한다.
4. 20분: 세 게이트의 “아니오/모름”을 해결 과제로 바꾼다.
5. 10분: 30일 안에 확인할 소유자와 증거를 배정한다.

점수 차이는 실패가 아니라 가치 있는 발견이다. 현업은 문서가 “다 있다”고 생각하고,
IT는 위치를 알지만 의미를 모르며, 보안은 복제 경로를 처음 들을 수 있다.

## 진단 결과물

- 유즈케이스 경계와 제외 범위
- 원천별 현재 점수와 증거 링크
- 수집·파일럿·운영 게이트의 중단 항목
- 30일 안에 해결할 상위 5개 과제
- 다음 재진단일과 책임자

진단이 끝났다면 [90일 로드맵](90-day-roadmap.md)에서 중단 항목을 먼저 일정에
넣는다. 기술 PoC를 먼저 넣지 않는다.
