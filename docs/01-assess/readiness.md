# Data Ready·Knowledge Ready 준비도 진단

진단의 목적은 높은 평균 점수가 아니다. 선택한 유즈케이스를 막는 **가장 낮은
준비축과 모르는 사실**을 찾는 것이다. Data Ready가 4점이고 Knowledge Ready가
1점이라면 평균 2.5점이 아니라 **지식 준비 1점**이 파일럿의 경계다.
[GAO AI 책임 프레임워크](https://www.gao.gov/products/gao-21-519sp)도
AI 책임을 거버넌스, 데이터, 성능, 모니터링이라는 서로 연결된 원칙으로 다룬다.

## 먼저 진단 범위를 한 문장으로 쓴다

> `[사용자]`가 `[업무 순간]`에 `[승인된 원천]`을 근거로 `[결정/행동]`하도록 돕는다.

“전사 지식 챗봇”처럼 범위가 없는 문장은 진단할 수 없다. 예를 들어 “설비 엔지니어가
고장 조치 전에 승인된 정비 절차와 최근 유사 장애를 찾아 점검 순서를 결정하도록
돕는다”라면 사용자, 순간, 원천, 행동이 모두 보인다.

## 두 준비축의 8개 진단 항목

각 항목은 증거가 있을 때만 점수를 준다. 인터뷰 답변만 있고 목록·정책·로그·샘플이
없다면 한 단계 낮게 기록한다.

| 준비축 | 진단 항목 | 핵심 질문 | 증거 예시 |
| --- | --- | --- | --- |
| Data | 원천·접근 | 필요한 원천·소유자·수집 권한을 아는가? | 원천 인벤토리, 접근 승인 |
| Data | 기계 판독 | 문서·표·메일·음성을 구조 손실 없이 읽는가? | 파싱 결과, OCR/ASR 표본 검사 |
| Data | 품질·표준 | 형식·단위·코드·결측·중복 규칙이 있는가? | 품질 규칙, 프로파일링 결과 |
| Data | 보호·계보 | ACL·버전·보존·삭제를 파생물까지 잇는가? | 원본 ID·해시, 계보, 삭제 로그 |
| Knowledge | 용어·의미 | 표준 정의·동의어·단위가 합의됐는가? | 업무 용어사전, 코드 매핑 |
| Knowledge | SSoT·효력 | 권위 원천·상태·시점·범위를 누가 승인하는가? | SSoT 대장, 승인·대체 기록 |
| Knowledge | 맥락·관계 | 대상·조건·예외와 관련 규칙을 연결하는가? | 지식 단위, 관계·규칙 목록 |
| Knowledge | 근거·평가 | 주장과 원문을 잇고 실제 질문으로 검증하는가? | 인용 위치, 승인 기록, 골든셋 |

## 0~5 성숙도

| 수준 | Data Ready에서 보이는 상태 | Knowledge Ready에서 보이는 상태 |
| --- | --- | --- |
| 0 | 원천·형식·권한을 모름 | 사람 기억과 암묵적 판단에 의존 |
| 1 | 위치와 담당자를 일부 목록화 | 핵심 용어·권위 후보를 발견 |
| 2 | 수집·보존·ACL·품질 책임을 통제 | 소유자가 용어·SSoT·효력 판정 |
| 3 | 추출·정규화·검사를 반복 실행 | 지식 단위·맥락·근거·승인을 표준화 |
| 4 | 변경·삭제가 추적되는 서비스 데이터 | 권한 범위에서 근거 답변·충돌·거절 가능 |
| 5 | 품질·계보·비용을 측정하고 개선 | 오류·정정·사용 피드백으로 지속 개선 |

::: warning 평균의 함정
보호·계보 또는 SSoT·효력이 0점인데 나머지가 5점이어도 파일럿을 시작하면 안 된다.
두 준비축을 하나의 평균으로 합쳐 위험한 결손을 감추지 않는다.
:::

## 네 개의 게이트

### 게이트 0 — 시작해도 되는가

다음 중 하나라도 “아니오/모름”이면 데이터를 복사하거나 모델에 넣지 않는다.

- [ ] 유즈케이스와 허용된 사용 목적이 문서화됐다.
- [ ] 원천의 업무 소유자와 시스템 관리자가 확인됐다.
- [ ] 개인정보·영업비밀·규제정보 등 정보등급을 분류했다.
- [ ] 처리·보존·삭제 근거와 접근 승인권자가 확인됐다.
- [ ] 외부 통신, 원격 측정, 업데이트 경로를 보안 담당자가 승인했다.

### 게이트 D — Data Ready인가

- [ ] 대표 문서와 예외·낡은 버전·충돌 문서를 포함한 샘플이 있다.
- [ ] 원본과 파생물의 ID·버전·해시·ACL을 연결한다.
- [ ] 문서·표·메일·음성의 구조 손실과 추출 오류를 검사한다.
- [ ] 형식·단위·코드·결측·중복의 데이터 품질 규칙이 있다.
- [ ] 변경·권한 회수·보존 만료·삭제를 반복 처리할 수 있다.

### 게이트 K — Knowledge Ready인가

- [ ] 핵심 용어의 표준 정의·동의어·소유자가 있다.
- [ ] SSoT 승인 기준과 “확정 불가” 처리 방식이 있다.
- [ ] 상태·효력 기간·적용 대상·조건·예외를 표현한다.
- [ ] 사실·규칙·결정이 원문 위치와 버전으로 추적된다.
- [ ] 정답, 인용, 명확화, 거절, 권한 차단을 검사할 골든셋이 있다.
- [ ] 사람 검토자와 오류 접수·수정 절차가 있다.

### 게이트 A — AI를 운영할 수 있는가

- [ ] 정확성뿐 아니라 검색 재현율, 인용 적합성, 거절 품질을 측정한다.
- [ ] 원본 변경·권한 회수·보존기간 만료가 색인과 캐시에 전파된다.
- [ ] 장애·오염·권한 누출을 탐지하고 격리·복구할 수 있다.
- [ ] 품질 SLO, 담당자, 검토 주기, 중단 기준이 합의됐다.

## 진단 워크숍 운영법

90분 워크숍에 현업 소유자, 기록·법무/개인정보, 보안, 시스템 운영, 데이터/AI
담당자가 함께 참석한다.

1. 10분: 유즈케이스 문장과 금지 범위를 합의한다.
2. 25분: 각자 [준비도 진단표](../templates/readiness-assessment.md)를 채운다.
3. 25분: 점수 차이가 2점 이상인 항목만 증거를 보며 토론한다.
4. 20분: 네 게이트의 “아니오/모름”을 해결 과제로 바꾼다.
5. 10분: 30일 안에 확인할 소유자와 증거를 배정한다.

점수 차이는 실패가 아니라 가치 있는 발견이다. 현업은 문서가 “다 있다”고 생각하고,
IT는 위치를 알지만 의미를 모르며, 보안은 복제 경로를 처음 들을 수 있다.

## 진단 결과물

- 유즈케이스 경계와 제외 범위
- 원천별 현재 점수와 증거 링크
- 시작·Data·Knowledge·AI 운영 게이트의 중단 항목
- 30일 안에 해결할 상위 5개 과제
- 기준선 업무 지표와 효과 소유자
- 다음 재진단일과 책임자

진단이 끝났다면 [경영효과 측정](business-impact.md)에서 기준선과 비교 방법을 정하고,
[90일 로드맵](90-day-roadmap.md)에 중단 항목을 먼저 넣는다. 기술 PoC를 먼저 넣지
않는다.
